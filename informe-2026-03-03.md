## 1. Investigación: Alertas "EasyCancha SIN cancha disponible"

### Problema reportado

Se detectaron alertas de tipo `easycancha_manual` indicando que no había cancha disponible para reservas provenientes de EasyCancha. Alertas recientes afectadas:

| Fecha | Hora | Cliente | Código EasyCancha |
|-------|------|---------|-------------------|
| 2026-03-06 | 21:00 | Gian Carlos Salinas Leyva | 26858423 |
| 2026-03-05 | 20:00 | Danilo Necul | 26855730 |
| 2026-03-02 | 21:00 | Carlos Aguilera | 26845348 |
| 2026-03-03 | 20:00 | Alejandro Nuñez | 26828932 |

Todas correspondían a **Lo Prado – Futbolito – horarios peak (20:00-21:00)**.

### Investigación realizada

Se revisaron:
- Base de datos Supabase (tablas `reservas`, `slots`, `alertas`)
- Logs de ejecución de n8n (workflow WF3 v2)
- Workflow del generador de slots
- Código fuente del dashboard

### Hallazgos

#### Caso Danilo Necul (2026-03-05, 20:00) — ALERTA CORRECTA

Las 6 canchas de Lo Prado estaban efectivamente ocupadas al momento de la alerta. La reserva fue resuelta manualmente vía dashboard, asignando Cancha 3 después de que otra reserva fuera cancelada.

#### Caso Gian Carlos Salinas Leyva (2026-03-06, 21:00) — ANOMALÍA

Al momento de la alerta (00:31), solo 3 de 6 canchas tenían reservas. Canchas 3, 4 y 5 deberían haber estado disponibles. Sin embargo, la query SQL del workflow retornó vacío. 30 minutos después, el mismo workflow encontró Cancha 3 exitosamente para otro cliente.

**Causa probable**: Error transitorio de base de datos o condición de carrera no reproducible.

**Dato clave descubierto**: El email de EasyCancha incluye el número de cancha asignada (campo `cancha_easycancha`), pero el workflow no lo utilizaba.

### Acciones correctivas ejecutadas

#### 1. Reserva manual creada

Se creó la reserva pendiente para Gian Carlos Salinas Leyva:
- **Cancha 1**, Lo Prado, 2026-03-06 21:00
- Código EasyCancha: 26858423
- Slot marcado como reservado
- Alerta marcada como leída

#### 2. Query SQL mejorada (preferencia de cancha)

**Antes**: Si EasyCancha indicaba una cancha específica, el workflow SOLO buscaba esa cancha. Si estaba ocupada, generaba alerta manual sin intentar otras.

**Después**: El workflow busca en TODAS las canchas disponibles, pero **prefiere** la cancha indicada por EasyCancha mediante `ORDER BY CASE`. Si la cancha preferida está ocupada, asigna cualquier otra disponible automáticamente.

#### 3. Mecanismo de retry agregado

**Antes**: Si no encontraba cancha disponible → alerta manual inmediata.

**Después**: Si no encuentra cancha → **espera 30 segundos** → reintenta la búsqueda → solo si sigue sin encontrar → alerta manual.

Esto cubre errores transitorios de base de datos como el caso de Gian Carlos.

**Nodos agregados al workflow WF3 v2:**
- `Wait 30s Retry` — pausa de 30 segundos
- `Retry Buscar Cancha` — segunda búsqueda SQL
- `Retry Cancha Encontrada?` — evaluación del segundo intento

#### 4. Workflow sincronizado

El archivo `workflows/WF3-v2-con-mod-cancel.json` del repositorio fue actualizado con la versión actual del workflow en n8n (42 nodos), incluyendo los nodos existentes `Check Duplicado` y `Ya Existe?` que no estaban en el repo, además de los 3 nuevos nodos de retry.

## 2. Integración de Alertas a Telegram

### Objetivo

Enviar todas las alertas del sistema al grupo de Telegram "Alertas Mundo 10" para que la secretaría reciba notificaciones en tiempo real sin necesidad de estar revisando el dashboard.

### Arquitectura implementada

```
Supabase (INSERT en tabla alertas)
    ↓ Database Trigger (pg_net)
n8n Webhook (WF-Telegram-Alertas)
    ↓ Formatear mensaje por tipo
Telegram Bot → Grupo "Alertas Mundo 10"
```

**Ventaja**: Un solo punto de control. Cualquier alerta insertada desde cualquier fuente (WF3 EasyCancha, bot multicanal, dashboard, futuros workflows) se envía automáticamente a Telegram sin modificar los workflows existentes.

### Componentes creados

#### Bot de Telegram
- Bot token configurado
- Grupo "Alertas Mundo 10" creado (chat ID: `-5206239395`)
- Mensaje de conexión exitosa enviado al grupo

#### Workflow n8n: `WF-Telegram-Alertas`
- **ID**: `4S3VSIzBrRPy0R6q`
- **Estado**: Activo
- **Nodos**: Webhook Trigger → Formatear Mensaje (Code) → Enviar a Telegram

Formato de mensajes según tipo de alerta:

| Tipo | Formato | Prioridad |
|------|---------|-----------|
| `easycancha_manual` | ⚠️ **ACCIÓN REQUERIDA** ⚠️ | Alta |
| `easycancha_error` | ❌ **ERROR EASYCANCHA** | Alta |
| `escalamiento` | 🔥 **ESCALAMIENTO** | Alta |
| `error` | 🚨 **ERROR SISTEMA** | Alta |
| `easycancha_sync` | ✅ **Sync EasyCancha OK** | Informativa |
| `reserva` | 📋 **Nueva Reserva** | Informativa |

#### Credencial Telegram en n8n
- **Nombre**: "Telegram Bot Alertas Mundo10"
- **ID**: `jYXeWt14hM76q7EX`

#### Trigger en Supabase
- Función `notify_telegram_on_alert()` creada con extensión `pg_net`
- Trigger `alertas_telegram_trigger` en tabla `alertas` (AFTER INSERT)
- Envía POST al webhook de n8n con los datos de la alerta

### Prueba end-to-end

Se insertó una alerta de prueba en Supabase. El trigger disparó el webhook de n8n en ~1 segundo, y el mensaje llegó correctamente al grupo de Telegram con el formato esperado. La alerta de prueba fue eliminada después de verificar.
